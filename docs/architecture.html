💚 Frontend — Vue 3 frontend/src/
📁src/
📄main.js imain.jsPoint d'entrée de l'application. Monte Vue, branche Pinia et Vue Router, puis attache tout au #app du HTML. entrypoint
📄App.vue iApp.vueComposant racine. Affiche la vue courante via <RouterView> et le système de toasts globaux (AppToast). root
📁router/
📄index.js irouter/index.jsDéclare toutes les routes de l'admin (/login, /dashboard, /users…). Contient les guards : redirige vers /login si non connecté, bloque /admins si pas super_admin. guards auth
📁stores/Pinia
📄auth.js istores/auth.jsGère l'état de connexion global : token Bearer, données de l'utilisateur connecté, actions login/logout/initAuth. Persiste le token dans localStorage.
📁services/Axios
📄api.js iservices/api.jsInstance Axios partagée. Injecte automatiquement le header Authorization: Bearer sur chaque requête. Intercepte les 401 → redirige vers /login. interceptors
📄authService.js iauthService.jsFonctions login(), logout(), me(). Appelle les endpoints /auth/* du backend.
📄userService.js iuserService.jsRegroupe les appels API pour les utilisateurs particuliers, les prestataires et les comptes admin : list, get, ban, activate, updateStatus…
📄categoryService.js icategoryService.jsCRUD des catégories de prestations + appel /toggle pour activer/désactiver sans recharger la page.
📄prestationService.js iprestationService.jsCRUD des prestations + updateStatus pour changer le statut (draft→published→suspended…) sans tout recharger.
📄eventService.js ieventService.jsCRUD des événements + updateStatus. Gère les filtres date et statut.
📁layouts/
📄AuthLayout.vue iAuthLayout.vueMise en page pour les pages non connectées (fond gris, carte centrée, logo UpcycleConnect). Utilisé uniquement par LoginPage. login
📄AdminLayout.vue iAdminLayout.vueMise en page principale du back-office : sidebar à gauche + header en haut + zone de contenu à droite. Toutes les pages admin s'affichent à l'intérieur. sidebar + header
📁components/layout/
📄TheSidebar.vue iTheSidebar.vueMenu de navigation vertical (fond sombre). Contient les liens Dashboard, Utilisateurs, Prestataires, Catégories, Prestations, Événements. Le lien Administrateurs n'apparaît que pour les super_admin.
📄TheHeader.vue iTheHeader.vueBarre du haut : affiche le titre de la page courante et le bouton "Se déconnecter" avec le nom de l'admin connecté.
📄NavItem.vue iNavItem.vueLien de navigation réutilisable dans la sidebar. Met en surbrillance automatiquement le lien actif selon la route courante.
📁components/common/
📄AppBadge.vue iAppBadge.vueÉtiquette colorée pour les statuts. Mappe automatiquement les valeurs : active/approved/published → vert, pending/draft → orange, banned/suspended → rouge.
📄AppButton.vue iAppButton.vueBouton réutilisable avec 4 variantes (primary, secondary, danger, ghost), gestion du loading (spinner) et état disabled.
📄AppPagination.vue iAppPagination.vueComposant de pagination générique. Affiche Précédent / numéros / Suivant. Émet un événement page-change que les pages écoutent pour recharger les données.
📄AppConfirmDialog.vue iAppConfirmDialog.vueBoîte de dialogue de confirmation (overlay sombre + modale). Utilisée avant toute action destructive : bannir, supprimer, désactiver.
📄AppToast.vue iAppToast.vueNotifications temporaires en bas à droite de l'écran (succès vert, erreur rouge, info bleu). Disparaissent automatiquement après 3-5 secondes.
📁pages/
📄auth/LoginPage.vue iLoginPage.vuePage de connexion admin. Formulaire email + mot de passe. Appelle authStore.login() → redirige vers /admin/dashboard si succès. Affiche les erreurs 401/403.
📄dashboard/DashboardPage.vue iDashboardPage.vueTableau de bord principal. Charge GET /dashboard/stats et affiche 5 cartes (utilisateurs, prestataires, en attente, prestations, événements). Alerte si prestataires en attente.
📄users/UserListPage.vue iUserListPage.vueListe paginée des utilisateurs particuliers. Recherche par nom/email, filtre par statut. Actions rapides : bannir ou activer directement depuis le tableau.
📄users/UserDetailPage.vue iUserDetailPage.vueFiche complète d'un utilisateur. Affiche ses informations et permet de le bannir ou l'activer avec confirmation.
📄providers/ProviderListPage.vue iProviderListPage.vueListe des prestataires filtrée par statut (en attente / approuvé / suspendu). Actions rapides inline pour approuver ou suspendre sans quitter la liste.
📄providers/ProviderDetailPage.vue iProviderDetailPage.vueFiche prestataire avec profil entreprise (SIRET, description, site web). Boutons Approuver / Suspendre avec mise à jour immédiate du badge statut.
📄admins/AdminListPage.vue iAdminListPage.vueListe des comptes administrateurs. Visible seulement pour les super_admin. Protège l'admin courant (pas de bouton supprimer sur son propre compte).
📄admins/AdminFormPage.vue iAdminFormPage.vueFormulaire création/édition d'un admin. Champs : prénom, nom, email, mot de passe, rôle (admin ou super_admin). En mode édition, l'email est désactivé.
📄categories/CategoryListPage.vue iCategoryListPage.vueListe toutes les catégories (actives et inactives). Toggle actif/inactif directement dans le tableau sans rechargement. Boutons Modifier et Supprimer.
📄categories/CategoryFormPage.vue iCategoryFormPage.vueFormulaire création/édition catégorie. Le slug est auto-généré depuis le nom (espaces → tirets, minuscules) mais reste modifiable manuellement.
📄prestations/PrestationListPage.vue iPrestationListPage.vueListe paginée des prestations avec filtres statut et catégorie. Changement de statut rapide (Publier / Suspendre) inline. Affiche le prestataire associé.
📄prestations/PrestationFormPage.vue iPrestationFormPage.vueFormulaire création/édition prestation. Sélecteurs dynamiques : catégorie (toutes) et prestataire (uniquement les approuvés). Prix optionnel selon le type de tarif.
📄events/EventListPage.vue iEventListPage.vueListe paginée des événements avec filtre statut. Changement de statut rapide (Publier / Annuler). Affiche lieu, date de début, nombre de places.
📄events/EventFormPage.vue iEventFormPage.vueFormulaire événement avec date pickers natifs (datetime-local). Validation front : la date de fin doit être après la date de début. Catégorie optionnelle.
🐘 Backend — Laravel 11 backend/
📁routes/
📄api.php iroutes/api.phpDéclare toutes les routes de l'API admin sous /api/admin/v1. Groupe les routes protégées derrière auth:sanctum + middleware admin. Groupe /admins derrière super_admin. toutes les routes
📁bootstrap/
📄app.php ibootstrap/app.phpPoint de démarrage de Laravel 11. Configure le routing API, enregistre les middlewares personnalisés (admin, super_admin), et personnalise les réponses d'erreur en JSON.
📁app/Http/Controllers/Admin/
📄AuthController.php iAuthController.phpGère login (vérifie email+password, vérifie le rôle admin, crée un token Sanctum 8h), logout (révoque le token), et me (retourne l'utilisateur connecté). login/logout/me
📄DashboardController.php iDashboardController.phpCalcule et retourne les statistiques globales : nombre d'utilisateurs, de prestataires (total + en attente), de prestations par statut, et d'événements actifs. stats
📄UserController.php iUserController.phpGestion des utilisateurs particuliers : liste paginée avec filtres, fiche, modification, bannissement, activation, suppression (soft delete). Enregistre les actions dans audit_logs.
📄ProviderController.php iProviderController.phpGestion des prestataires : liste filtrée par statut, fiche avec profil entreprise, changement de statut (pending/approved/suspended) avec enregistrement de approved_by et approved_at.
📄AdminUserController.php iAdminUserController.phpRéservé aux super_admin. Crée, modifie et supprime les comptes administrateurs. Protège contre la suppression de son propre compte. super_admin
📄CategoryController.php iCategoryController.phpCRUD complet des catégories + action toggle pour basculer is_active. Auto-génère le slug depuis le nom si absent. Audit log sur chaque action.
📄PrestationController.php iPrestationController.phpCRUD prestations avec filtres (statut, catégorie, prestataire). Action updateStatus séparée pour changer le statut seul sans tout revalider.
📄EventController.php iEventController.phpCRUD événements. Enregistre automatiquement created_by = admin connecté. Filtres par statut et plage de dates. Action updateStatus séparée.
📁app/Http/Middleware/
📄EnsureIsAdmin.php iEnsureIsAdmin.phpMiddleware RBAC. Vérifie que l'utilisateur authentifié a le rôle "admin" ou "super_admin". Retourne 403 JSON sinon. Appliqué à toutes les routes protégées. RBAC
📄EnsureIsSuperAdmin.php iEnsureIsSuperAdmin.phpMiddleware RBAC strict. Vérifie uniquement le rôle "super_admin". Appliqué aux routes /admins (création/suppression de comptes admin). RBAC
📁app/Http/Requests/Admin/
📄LoginRequest.php iLoginRequest.phpValide les données de connexion côté serveur : email requis et valide, mot de passe requis minimum 8 caractères. Retourne 422 si invalide.
📄StoreCategoryRequest.php iStoreCategoryRequest.phpValide la création et la modification de catégories. Vérifie l'unicité du slug en base (sauf pour la catégorie en cours de modification).
📄StorePrestationRequest.php iStorePrestationRequest.phpValide les prestations : provider_id doit exister en base, prix numérique positif, statut parmi les valeurs autorisées, type de tarif valide.
📄StoreEventRequest.php iStoreEventRequest.phpValide les événements côté serveur. Règle clé : end_date doit être après start_date (règle "after:start_date" Laravel).
📄StoreAdminRequest.php iStoreAdminRequest.phpValide la création d'un admin : email unique en base, mot de passe 8 caractères minimum, rôle obligatoirement "admin" ou "super_admin".
📁app/Http/Resources/Admin/
📄UserResource.php iUserResource.phpTransforme un modèle User en JSON propre pour l'API. Expose uniquement les champs utiles (pas le mot de passe). Ajoute le "primary_role" de l'utilisateur.
📄ProviderResource.php iProviderResource.phpÉtend UserResource avec les données du profil prestataire : company_name, siret, description, website, statut de validation.
📄CategoryResource.php iCategoryResource.phpSérialise une catégorie. Inclut prestations_count (nombre de prestations liées) quand il est chargé via withCount().
📄PrestationResource.php iPrestationResource.phpSérialise une prestation avec ses relations category et provider imbriquées (chargées avec whenLoaded pour éviter les N+1).
📄EventResource.php iEventResource.phpSérialise un événement avec category et creator imbriqués. Les dates sont en format ISO 8601.
📁app/Models/
📄User.php iUser.phpModèle central. Gère tous les types d'utilisateurs (particuliers, prestataires, admins). Relations : roles (BelongsToMany), providerProfile (HasOne), prestations (HasMany). Helpers isAdmin(), isSuperAdmin().
📄Role.php iRole.phpModèle simple pour les rôles. Valeurs possibles : "admin" et "super_admin". Relation inverse vers les users via la table pivot user_roles.
📄ProviderProfile.php iProviderProfile.phpExtension du User pour les prestataires. Stocke company_name, siret, description, website et le statut de validation (pending/approved/suspended).
📄PrestationCategory.php iPrestationCategory.phpCatégorie de prestation avec auto-génération du slug dans le boot(). SoftDeletes activé. Relations vers prestations et events.
📄Prestation.php iPrestation.phpPrestation de service liée à un prestataire (provider_id) et optionnellement à une catégorie. SoftDeletes. Statuts : draft, published, suspended, archived.
📄Event.php iEvent.phpÉvénement de la plateforme. Utilise la table "platform_events" (nom choisi pour éviter le conflit avec le mot réservé SQL "events"). SoftDeletes. Dates castées en datetime. table: platform_events
📄AuditLog.php iAuditLog.phpEnregistrement immuable des actions sensibles. Stocke action (ex: "user.banned"), resource_type, resource_id, et les valeurs avant/après en JSON pour traçabilité.
📁app/Services/Admin/
📄AuditService.php iAuditService.phpService injecté dans les controllers. Méthode log() : récupère automatiquement l'admin connecté et son IP, puis crée un enregistrement dans audit_logs. Appelé sur : ban, approve, delete, createAdmin. logs actions

📁database/seeders/données démo
📄RoleSeeder.php iRoleSeeder.phpCrée les 2 rôles : "admin" et "super_admin" dans la table roles.
📄AdminUserSeeder.php iAdminUserSeeder.phpCrée les comptes de démo : superadmin@upcycleconnect.fr (super_admin) et admin@upcycleconnect.fr (admin). Mots de passe hashés avec bcrypt.
📄DemoDataSeeder.php iDemoDataSeeder.phpGénère toutes les données de démonstration : 4 catégories, 10 utilisateurs, 5 prestataires (3 approuvés + 2 en attente), 8 prestations, 4 événements.
🗄️ Base de données MySQL 8
📋users iTable usersTable centrale pour TOUS les types d'utilisateurs (particuliers, prestataires, admins). Le rôle est déterminé par la table user_roles. SoftDelete avec deleted_at. softdelete
id, email, password (bcrypt), first_name, last_name, phone, status
📋roles iTable rolesContient uniquement 2 lignes : "admin" et "super_admin". Liée à users via la table pivot user_roles.
📋user_roles iTable user_rolesTable pivot Many-to-Many entre users et roles. Clé primaire composite (user_id, role_id). Un user sans entrée ici est un utilisateur particulier. pivot
📋provider_profiles iTable provider_profilesExtension de users pour les prestataires. user_id est UNIQUE (1 profil par user). Stocke company_name, siret, description, website et le statut de validation avec qui a approuvé et quand.
company_name, siret, status: pending/approved/suspended
📋prestation_categories iTable prestation_categoriesCatégories (Mobilier, Textile…). Slug unique auto-généré. is_active pour activer/désactiver. sort_order pour l'ordre d'affichage. SoftDelete. softdelete
name, slug (UNIQUE), is_active, sort_order
📋prestations iTable prestationsServices proposés par les prestataires. Liée à users (provider_id) et prestation_categories. SoftDelete. Statuts : draft, published, suspended, archived. softdelete
title, price (decimal), price_type, status
📋platform_events iTable platform_eventsNommée "platform_events" et non "events" pour éviter le conflit avec le mot réservé SQL. Stocke start_date et end_date en DATETIME. created_by pointe vers l'admin qui a créé l'événement. softdelete
start_date, end_date, location, max_participants
📋audit_logs iTable audit_logsJournal immuable des actions sensibles. Enregistre qui a fait quoi, sur quelle ressource, avec les valeurs avant et après en JSON. Pas de updated_at, pas de softdelete : ces logs ne se modifient jamais. traçabilité
action, resource_type, old/new_values (JSON)
📋personal_access_tokens iTable personal_access_tokensTokens Bearer gérés par Laravel Sanctum. Chaque connexion admin crée un token (expires_at = +8h). Révoqué à la déconnexion. Le hash du token est stocké, jamais le token en clair. Sanctum

image: nginx:1.25-alpine
→ /api/admin/ → backend:80
→ / → frontend:80

Node 20 → npm build → Nginx static
📦backend iService backendPHP 8.3 avec Apache. Reçoit les requêtes /api/* de Nginx. Les variables d'environnement (DB_HOST=mysql, APP_KEY…) sont injectées depuis le .env. Volumes pour storage et logs persistants. interne :80
PHP 8.3 + Apache + mod_rewrite
volumes: storage/ + logs/ (persistants)
📦mysql iService mysqlMySQL 8. Non exposé à l'extérieur. Accessible uniquement depuis le réseau interne Docker sous le nom d'hôte "mysql". Volume mysql_data pour que les données survivent aux redémarrages. interne :3306
volume mysql_data (données persistantes)
healthcheck → backend démarre après MySQL

Fichiers de config

📄.env i.env (racine)Variables d'environnement injectées dans tous les conteneurs au démarrage. Contient APP_KEY, mots de passe BDD, domaines CORS. Ne jamais committer ce fichier en git. variables runtime

Auth flow (Bearer token)

1.POST /auth/login → token Bearer
2.Token stocké localStorage
3.Header Authorization sur chaque req
4.Middleware EnsureIsAdmin vérifie
5.Expiration : 8 heures
6.401 → logout auto + redirect /login
